# Architecture

## Overview

`rphantom` is a linear pipeline of six modules. Each stage consumes the previous
stage's output, is independently testable, and is deterministic given its inputs.

```
                 ┌──────────────┐
   seed, params  │  phantom.py  │  Phantom(volume, mask, spacing, ground_truth)
 ───────────────▶│  (IBSI-free) │──────────────┐
                 └──────────────┘               │
                                                ▼
                 ┌──────────────┐      ┌─────────────────┐
   scanner knobs │acquisition.py│      │   features.py   │  169-D feature vector
 ───────────────▶│ blur, noise, │─────▶│ IBSI 3.1–3.11,  │──────────┐
                 │ resample, …  │ Acq. │ 482 ref. values │          │
                 └──────────────┘      └─────────────────┘          │
                                                                    ▼
                 ┌──────────────┐                          ┌─────────────────┐
                 │ normalize.py │◀─────────────────────────│  stability.py   │
                 │ invert the   │   calibration curve      │ ICC(2,1), CCC,  │
                 │ physics      │                          │ stability atlas │
                 └──────────────┘                          └─────────────────┘
```

The GUI (`apps/`) is a view over `phantom → acquisition → features`; `examples/`
and `scripts/` are runnable entry points; `tests/` validates every stage.

## Module responsibilities

| Module | Input | Output | Depends on |
|---|---|---|---|
| `phantom.py` | seed + texture/lesion parameters | `Phantom` dataclass | numpy, scipy |
| `acquisition.py` | `Phantom` + scanner parameters | `Acquisition` dataclass | numpy, scipy, `phantom` |
| `features.py` | volume, mask, spacing | feature dataclasses / tag→value dict | numpy, scipy, scikit-image |
| `stability.py` | phantoms × conditions | `StabilityAtlas` | numpy, `acquisition`, `features`, `phantom` |
| `normalize.py` | calibration samples | `CalibrationCurve` | numpy |

Note the dependency direction: `features` does **not** import `acquisition` or
`phantom`; it operates on plain arrays. This keeps the feature core reusable on
any volume, and keeps its 482-value IBSI validation independent of how the volume
was produced.

## Data conventions

- **Axis order** is `(z, y, x)` everywhere; **spacing** is `(dz, dy, dx)` in mm.
- **Volumes** are `float32` on a HU-like scale; **masks** are boolean.
- **Discretised levels** are integers `1..Ng`; absent levels still count (the IBSI
  phantom spans 6 levels with 2 empty).
- **Feature tags** are the official IBSI strings (e.g. `cm_contrast_3D_comb`), so
  results join directly against published tables.

## Load-bearing design decisions

**Gaussian random field texture.** The phantom background is white noise filtered
by an anisotropic Gaussian in the Fourier domain. This makes the correlation
length an exact, prescribed, *measurable* parameter — `measure_correlation_length`
recovers it — which is what lets texture be a controlled variable in stability
studies. Because the FFT convolution is circular, a correlation length beyond half
the field extent would wrap; the generator raises rather than return the wrong
texture.

**Feature core is IBSI-first and dependency-free.** Every family is implemented
straight from the IBSI definitions on `numpy`, not wrapped from PyRadiomics. The
sole source of truth for correctness is the IBSI digital-phantom reference table,
reproduced at zero tolerance. Fixtures are generated by
`scripts/fetch_ibsi_reference.py`, never hand-transcribed.

**All aggregations are first-class.** GLCM/GLRLM expose all six IBSI aggregations
and the zone families all three, because the *choice* of aggregation is itself a
source of feature variability the project exists to quantify. The default
feature vector uses the 3D-merged variants.

**Zones as lists, not dense matrices.** A dense GLSZM is `Ng × max_zone_size`,
and the largest zone can be the whole ROI. Zones are stored as a `(level, value)`
list instead — smaller, and closer to the summation the features actually are.

**ICC and CCC are our own.** `stability.py` implements ICC(2,1) from the two-way
ANOVA decomposition and Lin's CCC from its definition, validated against
`pingouin` in the tests. This keeps `pingouin`/`pandas` off the runtime import
path, so importing `rphantom` stays light.

**Physics-based, not cohort-based, normalisation.** Rather than harmonise against
a reference dataset (ComBat), `normalize.py` learns a feature's response to a
*known* acquisition descriptor on the phantom and inverts it. A feature whose
response the model cannot capture (`R² < 0.9`) is refused, not silently corrected.

## Failure model

There is one rule: **never return a silent `nan`**. Each module raises a specific
`ValueError` subclass with a message that says what to change:

| Exception | Raised by | Example |
|---|---|---|
| `FeatureError` | `features.py` | single grey level → GLCM correlation undefined |
| `AcquisitionError` | `acquisition.py` | resample target coarser than the ROI |
| `StabilityError` | `stability.py` | ICC with fewer than two conditions |
| `NormalizationError` | `normalize.py` | calibration descriptor has no variation |

## Repository layout

```
rphantom/        the core library (six modules)
apps/            Phantom Studio GUI (view) + studio_core (headless logic)
examples/        runnable scripts: slice render, acquisition sweep, full atlas
scripts/         fetch_ibsi_reference.py — regenerate test fixtures from source
tests/           716 tests: IBSI validation, contracts, determinism, GUI logic
docs/            this documentation
```
